# Built-in Bloom Filter Index Access Method

## Summary

Promote PostgreSQL's contrib `bloom` index access method into a first-class,
built-in index AM in YugabyteDB. Users can create bloom indexes without
installing an extension:

```sql
CREATE INDEX ON table_name USING bloom (col1, col2);
```

Like other non-DocDB index AMs (`hash`, `gin`, `brin`, `gist`, `btree`), bloom
is supported on temporary (and other non-YB-storage) relations. Attempts to use
it on YB storage relations fail with the existing
`index method "bloom" not supported yet` error (`yb_amisforybrelation = false`).

## Motivation

- Bloom indexes are useful for multi-column equality filters with a compact
  signature representation and bitmap scans.
- Shipping bloom only as `CREATE EXTENSION bloom` is inconsistent with how YSQL
  exposes other core index AMs and blocks `USING bloom` out of the box.
- Registering the AM in `pg_am.dat` matches the ybgin/lsm pattern and keeps
  catalogs generated by initdb self-contained.

## Non-Goals

- DocDB-backed `ybbloom` AM for permanent YB tables (future work, analogous to
  `ybgin`).
- Changing bloom's on-disk format or signature algorithm.
- Supporting unique, ordered, or index-only scans (bloom remains bitmap-only).

## Design

### Code layout

Move sources from `src/postgres/contrib/bloom/` to
`src/postgres/src/backend/access/bloom/`, with the public header at
`src/postgres/src/include/access/bloom.h`. Wire the directory into
`src/backend/access/Makefile` exactly like `brin`/`hash`/`ybgin`.

Adapt extension-only pieces for in-core linking:

- Drop `PG_FUNCTION_INFO_V1(blhandler)` (handler is a normal backend symbol).
- Replace `_PG_init` dynamic `add_reloption_kind()` registration with a static
  `RELOPT_KIND_BLOOM` and entries in `reloptions.c` (`length`, `col1`..`col32`).
- Set YB `IndexAmRoutine` fields explicitly (`yb_amisforybrelation = false`).
- Keep using `generic_xlog` (no new rmgr).

### Catalog registration

| Object | OID | Notes |
|--------|-----|--------|
| `pg_am.bloom` | 8500 (`BLOOM_AM_OID`) | handler `blhandler` |
| `pg_proc.blhandler` | 8501 | `index_am_handler` |
| opfamilies | 8502–8503 | `integer_ops`, `text_ops` |
| opclasses | 8504–8505 | default `int4_ops`, `text_ops` |
| `pg_amop` / `pg_amproc` | unnamed | strategy 1 `=`; support proc 1 hash function |

Initial opclasses mirror contrib `bloom--1.0.sql` (`int4`, `text`) so
`CREATE INDEX ... USING bloom (col1, col2)` works for the common case. More
types can be added later without format changes.

### Upgrade path

Add `src/yb/yql/pgwrapper/ysql_migrations/V100__builtin_bloom_access_method.sql`
inserting the same catalog rows with `ON CONFLICT DO NOTHING` and pinning
`pg_depend` entries, following `V8__7850__ybgin_access_method.sql`.

### Contrib extension

After the built-in AM exists, remove `bloom` from `contrib/Makefile` and delete
the extension sources so `CREATE EXTENSION bloom` is no longer offered (avoids
duplicate AM name). Document the replacement: use `USING bloom` directly.

### Testing

Add `yb.orig.bloom` regress tests (temp tables) covering:

1. Index creation (`USING bloom (col1, col2)` plus reloptions).
2. Query plans choosing bitmap index scans with recheck.
3. Correct results when false positives require heap recheck.
4. Multi-column equality filtering.
5. `VACUUM` / `REINDEX` leaving the index queryable and uncorrupted.
6. Rejection on permanent YB-storage tables.

Schedule via new `yb_bloom_schedule` included alongside other index suites.

## Key Decisions

1. **Built-in, not extension** — matches user requirement and other core AMs;
   removes `CREATE EXTENSION` friction.
2. **Temp-table support only (v1)** — reuses local page storage from contrib
   bloom; follows gin/hash/brin YB support model instead of inventing DocDB
   storage up front.
3. **Minimal opclass set (int4, text)** — enough for required tests and the
   documented `USING bloom (col1, col2)` example; avoids a huge catalog diff.
4. **Static `RELOPT_KIND_BLOOM`** — consistent with brin/gin/hash; avoids
   extension `_PG_init` in the backend.
5. **Stacked PRs** — small, reviewable commits with clear dependencies.

## Open Questions

None blocking implementation. A future `ybbloom` AM for DocDB tables is
explicitly out of scope.

## PR Plan

### PR 1: Move bloom AM sources into the backend

- **Files/components affected:** `src/postgres/src/backend/access/bloom/*`,
  `src/postgres/src/include/access/bloom.h`,
  `src/postgres/src/backend/access/Makefile`,
  `src/postgres/src/include/access/reloptions.h`,
  `src/postgres/src/backend/access/common/reloptions.c`
- **Dependencies:** None
- **Description:** Copy and adapt contrib bloom into `backend/access/bloom`,
  add `RELOPT_KIND_BLOOM` and static reloptions, integrate into the backend
  build. No catalog entries yet (AM not visible to SQL).

### PR 2: Register bloom in system catalogs and upgrade migration

- **Files/components affected:** `pg_am.dat`, `pg_proc.dat`, `pg_opfamily.dat`,
  `pg_opclass.dat`, `pg_amop.dat`, `pg_amproc.dat`,
  `ysql_migrations/V100__builtin_bloom_access_method.sql`
- **Dependencies:** PR 1
- **Description:** Register `bloom` in `pg_am` with handler `blhandler`, add
  int4/text opfamilies and opclasses, and ship a YSQL migration for existing
  clusters.

### PR 3: Regression tests for built-in bloom indexes

- **Files/components affected:** `src/postgres/src/test/regress/sql/yb.orig.bloom.sql`,
  `expected/yb.orig.bloom.out`, `yb_bloom_schedule`
- **Dependencies:** PR 2
- **Description:** Cover index creation, plans, false-positive recheck,
  multi-column filters, VACUUM/REINDEX integrity, and YB-storage rejection.

### PR 4: Remove contrib bloom extension

- **Files/components affected:** `src/postgres/contrib/bloom/**`,
  `src/postgres/contrib/Makefile`
- **Dependencies:** PR 3
- **Description:** Drop the extension package now that the AM is built-in, avoiding
  duplicate `bloom` access method installation.
